# CNC 工厂订单管理 Web 应用设计

## 背景

本项目从空仓库开始，目标是构建一个可本地部署的 CNC 加工工厂订单管理 Web 应用。第一版服务一个工厂账户，用于管理机器、甲方客户订单，以及每天从机器维度录入的加工量和出货量。

系统采用 MVP 范围，优先保证真实可用、数据关系清晰、后续可扩展为多账户/多工厂场景。

## MVP 目标

- 使用 `.env` 预置账号密码登录系统。
- 支持同一账户多端同时登录。
- 支持配置 CNC 机器，初始场景可添加 15 台机器，编号为 1 到 15。
- 支持创建订单，订单中直接填写客户/甲方工厂名称。
- 支持一台机器当前关联一个进行中订单。
- 支持从单台机器页面录入多次加工/出货记录。
- 支持订单页查看每个订单的计划数量、已加工、已出货、剩余数量、超计划提示和结单状态。
- 支持机器详情页查看每天和历史的加工/出货记录。
- 支持按日期、机器、客户/甲方工厂、订单、状态等常用条件筛选。
- 使用 Docker Compose 管理 `web` 和 `db` 两个容器。

## 非目标

第一版不实现多员工账号、角色权限、开放注册、排产、工序管理、库存、附件/图纸上传、Excel/CSV 导出、操作日志、自动结单、仓库独立出货和复杂报表。

这些功能不进入 MVP，但数据结构会保留扩展空间，避免后续多用户或多工厂改造时推翻核心模型。

## 技术架构

采用 `Next.js + PostgreSQL + Prisma`。

- `web` 容器：运行 Next.js 应用，包含页面、API、鉴权、服务端校验和 Prisma 数据访问。
- `db` 容器：运行 PostgreSQL。
- `docker compose`：管理两个容器、数据库持久卷、环境变量和启动顺序。

选择 PostgreSQL 而不是 MongoDB，原因是机器、订单、生产记录和出货记录是强关系数据，并且需要稳定地进行日期、客户、状态、订单进度等筛选和汇总。SQL 更适合这类约束和统计。

本机开发时，如果缺少 Docker 相关工具，实施阶段会先检测环境。WSL 场景优先使用 Docker Desktop 的 WSL 集成；如果环境允许，也可以在 WSL 内安装 Docker Engine 和 Compose 插件。

## 账户和鉴权

MVP 只有一个工厂账户，不做员工账号和角色权限。

账号来源：

- `.env` 提供初始用户名和密码。
- 应用启动或初始化脚本保证该账号存在。
- 密码只保存哈希，不保存明文。

登录规则：

- 登录成功后写入安全 cookie session。
- 允许同一账户在多台设备和多个浏览器中同时登录。
- 业务查询和写入都基于当前用户所属的 `workspace_id` 过滤。

扩展预留：

- 数据库保留 `workspaces` 和 `users`。
- 所有业务表携带 `workspace_id`。
- 后续增加开放注册、多账户或多员工权限时，可以复用隔离字段。

## 数据模型

### Workspaces

表示工厂工作区。MVP 中只有一个工作区。

关键字段：

- `id`
- `name`
- `created_at`
- `updated_at`

### Users

表示登录账号。MVP 中由 `.env` 初始化一个用户。

关键字段：

- `id`
- `workspace_id`
- `username`
- `password_hash`
- `created_at`
- `updated_at`

约束：

- 同一工作区内用户名唯一。

### Machines

表示 CNC 机器。

关键字段：

- `id`
- `workspace_id`
- `code`：机器编号，例如 `1`、`2`、`15`
- `name`：机器名称
- `model`：型号/规格
- `location`：位置
- `status`：机器状态
- `notes`：备注
- `current_order_id`：当前关联订单
- `created_at`
- `updated_at`

约束和规则：

- 同一工作区内机器编号唯一。
- 机器当前只能关联一个订单。
- 机器已有历史记录时不硬删除，改为停用或保留。

机器状态建议值：

- `active`：正常使用
- `idle`：空闲
- `maintenance`：维护中
- `disabled`：停用

### Orders

表示甲方客户订单。MVP 采用一单一种工件。

关键字段：

- `id`
- `workspace_id`
- `customer_name`：客户/甲方工厂名称
- `order_no`：订单号
- `part_name`：工件名称/型号
- `planned_quantity`：计划数量
- `due_date`：交期
- `status`：订单状态
- `notes`：备注
- `closed_at`：手动结单时间
- `created_at`
- `updated_at`

约束和规则：

- 计划数量必须为正整数。
- 客户名称、工件名称和计划数量为必填。
- 出货数量达到或超过计划数量时，界面提示“可结单”。
- 订单不会自动结单，必须由用户手动结单。
- 已有生产/出货记录的订单不硬删除，避免历史记录断链。

订单状态建议值：

- `open`：进行中
- `closed`：已结单

订单汇总字段不直接存储为可编辑值，而是由生产记录汇总得到：

- `completed_quantity`：累计加工数量
- `shipped_quantity`：累计出货数量
- `remaining_quantity`：计划数量减累计出货数量
- `is_over_planned`：加工或出货数量超过计划数量
- `can_close`：累计出货数量达到或超过计划数量，且订单未结单

### Production Records

表示从机器页面录入的一次加工/出货记录。

关键字段：

- `id`
- `workspace_id`
- `machine_id`
- `order_id`：记录创建时机器关联的订单快照
- `recorded_at`：记录时间
- `completed_quantity`：本次加工完成数量
- `shipped_quantity`：本次出货数量
- `notes`：备注
- `created_at`
- `updated_at`

约束和规则：

- 新增记录时机器必须已关联订单。
- 一天内同一台机器允许多次录入记录。
- 加工数量和出货数量允许为 0，但一条记录中至少有一个数量必须大于 0。
- 每条记录保存当时的 `order_id`，机器后续切换订单不影响历史。
- 记录允许编辑和删除。
- 编辑或删除记录后，机器详情和订单详情的汇总实时重算。
- 数量超过订单计划时允许保存，但界面显示超计划提示。

## 页面和交互

### 登录页

用户输入 `.env` 预置的账号密码登录。登录失败时给出明确提示，不暴露系统内部细节。

### 主布局

采用侧边栏工作台结构。

桌面端：

- 左侧固定导航：机器、订单、记录。
- 右侧为当前页面内容。
- 机器详情、订单详情和筛选区域优先使用桌面空间，提高长期录入效率。

手机端：

- 导航折叠为顶部或底部导航。
- 列表转为卡片。
- 详情和录入表单纵向排列。
- 保证能完成查看、筛选和简单录入。

### 机器页

机器页展示机器列表/卡片，并提供筛选和新增入口。

筛选条件：

- 机器编号
- 机器状态
- 当前关联订单

机器列表展示：

- 编号
- 名称
- 状态
- 当前关联订单
- 今日加工数量
- 今日出货数量

机器详情展示：

- 机器基础信息
- 当前关联订单
- 今日加工/出货汇总
- 历史生产/出货记录
- 新增记录表单
- 编辑机器信息
- 关联或切换订单

新增记录表单字段：

- 记录时间
- 加工数量
- 出货数量
- 备注

新增记录时自动绑定该机器当前关联的订单。机器未关联订单时，隐藏或禁用录入入口，并提示先关联订单。

### 订单页

订单页展示订单列表、筛选器、新增订单入口和订单详情入口。

筛选条件：

- 客户/甲方工厂名称
- 订单号
- 工件名称/型号
- 订单状态
- 交期范围

订单列表展示：

- 客户名称
- 订单号
- 工件名称/型号
- 计划数量
- 已加工数量
- 已出货数量
- 剩余数量
- 状态
- 超计划提示
- 可结单提示

订单详情展示：

- 订单基础信息
- 当前关联机器
- 历史生产/出货记录
- 汇总进度
- 手动结单
- 重新打开

订单手动结单后，不影响历史记录查询。重新打开订单后可以继续被机器关联和录入。

### 记录页

记录页作为查询和修正入口，展示所有生产/出货记录。

筛选条件：

- 日期范围
- 机器
- 客户/甲方工厂
- 订单
- 订单状态

列表展示：

- 记录时间
- 机器编号/名称
- 客户名称
- 订单号
- 工件名称/型号
- 加工数量
- 出货数量
- 备注

记录页支持编辑和删除记录。修改后，相关订单和机器汇总实时变化。

## 校验和错误处理

- 机器编号在同一工作区内必须唯一。
- 订单计划数量必须为正整数。
- 生产记录中的加工数量和出货数量不能为负数。
- 一条生产记录中加工数量和出货数量不能同时为 0。
- 机器未关联订单时不能新增生产记录。
- 订单或机器已有记录时不硬删除。
- 所有 API 写入操作都校验当前用户的 `workspace_id`。
- 登录失败、校验失败、资源不存在和越权访问返回明确但不过度暴露内部细节的错误信息。

## 测试策略

后端/API 测试重点：

- 登录成功和失败。
- `.env` 账号初始化。
- 工作区隔离。
- 机器编号唯一性。
- 新增、编辑和停用机器。
- 新增订单和手动结单。
- 机器关联订单。
- 机器未关联订单时禁止新增记录。
- 一天内同一机器多次录入。
- 记录编辑和删除后订单汇总重算。
- 超计划数量允许保存并可被查询到。

前端测试重点：

- 登录流程。
- 机器列表筛选和机器详情。
- 关联订单。
- 从机器详情新增生产/出货记录。
- 订单列表汇总展示和筛选。
- 记录页筛选、编辑、删除。
- 桌面和手机布局基本可用。

部署验证：

- 从空数据库启动 `docker compose up`。
- 自动完成数据库迁移。
- 初始化 `.env` 账号。
- 登录后可新增机器、订单和生产记录。

## 后续扩展方向

后续可在不推翻核心模型的基础上增加：

- 多工厂工作区。
- 多员工账号和角色权限。
- 操作日志。
- Excel/CSV 导入导出。
- 附件/图纸上传。
- 排产和工序。
- 库存和仓库独立出货。
- 异常停机和维护记录。
- 更完整的统计仪表盘。
